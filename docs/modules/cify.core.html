Core#

This page contains the documentation for core components of CIFY. The first three components, Algorithm, Agent and Collection are from the base_classes package.

The core package contains all core objects used across paradigms in CIFY.

The core package contains one subpackage, base_classes, that contains some base classes worth checking out. These classes, plus some remaining noteworthy classes are given in the list below.

  • Agent

  • Algorithm

  • Collection

  • ObjectiveFunction

  • MultiObjectiveFunction

  • Position

  • Runner

  • Task

The utilities module contains useful utilities that are not paradigm-specific. General topologies and visualizations are also contained in the core package.

Algorithm#

class Algorithm(obj_func: ObjectiveFunction, global_optimum_pos: Optional[Position] = None, name: Optional[str] = None, **kwargs)#

Bases: ABC

The Algorithm class is a base class used by all implemented algorithms. The class handles iterating, statistics and handling of fields used by all algorithms. This base class is used when implementing novel algorithms not supplied by CIFY, whilst maintaining integration with all the other CIFY components.

__init__(obj_func: ObjectiveFunction, global_optimum_pos: Optional[Position] = None, name: Optional[str] = None, **kwargs)#
Parameters:

  • obj_func (ObjectiveFunction) – The ObjectiveFunction this Algorithm will be optimizing.

  • global_optimum_pos (Position, optional) – The currently found global optimum. This parameter would usually not be set on init.

  • name (str, optional) – An optional name for the Algorithm.

abstract do_iteration()#

The only function that must be overridden when implementing your own algorithm. This function must be the logic of one iteration of your algorithm and must return a list of the collections used by your algorithm so that statistics can be calculated. It is important that this method returns a list of the Collection’s used by the algorithm at the end of the iteration. These will be used to update the statistics.

Returns:

A list of Collection s.

execute(n_iterations=1, n_evaluations=0, stopping_condition=None, log=False)#

A wrapper of the Algorithm.iterate() method.

iterate(n_iterations=1, n_evaluations=0, stopping_condition=None, log=False)#

Iterate the Algorithm.

Parameters:

  • n_iterations (int, optional) – The number of iterations to perform, defaults to 1.

  • n_evaluations (int, optional) – The number of evaluations to perform. If no value is set, the number of evaluations will not be used as a stopping criterion.

  • stopping_condition (callable, optional) – A callable that returns a boolean that determines when to terminate iterating over the algorithm. If a stopping condition is provided, it will override the n_iterations parameter.

  • log (bool, optional) – Whether to print to the console after completing each iteration.

Agent#

class Agent(name: Optional[str] = None, **kwargs)#

Bases: ABC, object

Agents are the individual elements in Collection objects. Any object can be an Agent simply by using Agent as a base class. There are three methods that must be implemented by any Agent class:

  • position(self)

  • p_best_position(self)

  • social_best_pos(self)

These methods define how the Agent will behave when used with general components such as topologies or evolutionary operators. A number of classes provided by CIFY use Agent as a base class, most notably Particle and Individual.

Warning

Not returning Position objects from these three methods will result in errors in the supplied algorithms.

Collection#

class Collection(agents: Optional[List[Agent]] = None, **kwargs)#

Bases: ABC

A Collection stores a list of Agent s with possible additional parameters. Storing Agent s in a Collection allows for containerising parameters that are to be used by all the Agent s in the Collection. For example, a swarm of particles, represented as a Collection of Particle ‘s, can have a custom velocity update function with its own set of parameters to be used in a multi-population optimization algorithm. All collection based algorithms such as those found in evolutionary computation and swam intelligence make use of Collection s of Agent s.

cmp(a, b) → bool#

A comparison method returned by an ObjectiveFunction which can be used when constructing algorithms to handle optimizing minimization or maximization objective functions. The cmp() method takes two arguments, the first argument is the value of a Position object that is being compared to a second Position object to see if it’s value is better. These arguments can be any real valued numbers.

For minimization objective functions the cmp() method checks whether the first position value is less than the second position value, and for maximization objective functions the cmp() method checks whether the first position value is greater than the second position value. Think of it as saying, “is this first value better than the second value”.

Parameters:

  • a (Any real valued number, typically a float) – The first value

  • b (Any real valued number, typically a float) – The second value

Returns:

Whether a is better than b.

Return type:

bool

evaluate(vector, function: Optional[int] = None) → float#

Evaluates a vector on one or more objective functions.

To evaluate a vector on all the objective functions:

evaluate(vector=<vector>)

This will return a list of the ObjectiveFunction values where the first item in the list is the value of the vector evaluated on the first ObjectiveFunction and the last item is the value of the vector evaluated on the last ObjectiveFunction passed to the MultiObjectiveFunction object.

To evaluate a vector on only one of the objective functions:

This will return a float representing the value of the vector evaluated on the ObjectiveFunction corresponding to the function parameter. The function parameter can be the index or name of a function.

Parameters:

  • function (int or str) – Which function to use for vector evaluation, given as an index or name.

Runner#

class Runner(tasks: Optional[List[Task]] = None, name: Optional[str] = None, log: bool = False)#

Bases: object

The Runner class is used to create a list of Task s to be run in sequential order. Task s can be manually created and added or removed from the Runner using the add and remove methods, however, it is not necessary to manually create Task s. Algorithm s and ObjectiveFunction’s can be added to or removed from the Runner using the add and remove methods from which the Runner will create Task s when its compile method is called.

  • log (bool, optional) – Whether to output a log at the end of each iteration, defaults to False.

compile(n_independent_runs=1, n_iterations=1, n_evaluations=0, stopping_condition=None, relationship='many-to-many')#

This is where all the conditions passed to __init__ and compile are transformed into a list of Task s that will be run when the user calls execute. Task s already in the tasks list remain unchanged. For more complex Task s, It is recommended that Task s are manually created and appended to the tasks list of the Runner instance. The conditions passed to this function apply to all tasks created from the algorithm_pool and objective_function_pool.

There are four relationship types: one-to-one, one-to-many, many-to-one and many-to-many. Each relationship type represents tasks are created when relating the Algorithm s in the algorithm_pool and the ObjectiveFunction s in the objective_function_pool.

Parameters:

  • n_independent_runs (int, optional) – The number of independent runs to perform for each Task, defaults to one.

  • n_iterations (int, optional but highly recommended) – The number of iterations to perform for each Task, defaults to one.

  • n_evaluations (int, optional) – The number of evaluations to terminate at for each Task.

  • stopping_condition (bool or callable that returns a bool, optional) – A stopping condition to apply to all Task s.

  • relationship (string, optional) – A string representing which cardinality to use when creating Task s, defaults to many-to-many.

summary() → str#

Returns a string formatted summary of the Runner s Task s to be executed.

Return type:

str

execute()#

Executes the Task s in the Runner s internal tasks list. If you have added Algorithm s and ObjectiveFunction s to this Runner using the add method, you should compile the Runner first.

Position#

class Position(vector: Optional[ndarray] = None, obj_func: Optional[ObjectiveFunction] = None, value: Optional[float] = None)#

Bases: object

A simple object that defines a position vector, and it’s evaluation. Position objects are typically handled by an instantiating Agent such as a Particle or Individual. It is rare that a user may wish to use Position objects directly.

Parameters:

  • vector (Numpy array or list) – The position vector. This will always be converted to a Numpy array.

  • obj_func (ObjectiveFunction or MultiObjectiveFunction) – The ObjectiveFunction to use for Position evaluation.

  • value (float or list of floats) – The evaluated value or values of the position vector.

property vector: ndarray#

Returns the vector as a Numpy array.

property obj_func: ObjectiveFunction#

Returns the Position’s ObjectiveFunction.

property value: float#

Returns the current value of the Position in its ObjectiveFunction’s search space.

If the value has already been evaluated and the Position has not changed its position, its vector has not changed, and the ObjectiveFunction has not changed, the value will not be re-evaluated. This is in an attempt to save from the computational budget of CIFY algorithms as no Position evaluations should take place when they do not need to.
